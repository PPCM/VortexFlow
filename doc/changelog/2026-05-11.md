# Changelog - 2026-05-11

## Summary

Vague 3 partielle : petits patches (fix fallback WebSocket, ajout
`.editorconfig`, suppression de 2 TODO obsolètes liés à la simulation
côté serveur retirée), puis mise en place de l'outillage DX au niveau
racine (Husky + lint-staged + Prettier).

## Changes

- [frontend/src/services/websocket.ts] Corrigé le fallback
  `VITE_WS_URL` qui pointait sur `http://localhost:3001` (port
  inexistant) — il pointe maintenant sur `http://localhost:5000`,
  qui est le port où le backend sert Socket.IO. Le frontend dev
  server (3000) n'a jamais hébergé de socket, donc l'ancien
  fallback ne pouvait jamais marcher en absence de `VITE_WS_URL`.
- [frontend/src/context/GraphContext.tsx] Supprimé 2 TODO et 2
  appels commentés obsolètes dans `updateSimulationConfig` :
  `apiService.updateSimulationConfig` et
  `webSocketService.updateSimulationConfig` ne seront jamais
  implémentés — la simulation tourne entièrement côté navigateur
  (ADR-002). Le commentaire qui remplace explique pourquoi la
  fonction reste juste un dispatch local.
- [.editorconfig] (new) Indentation 2 espaces, LF, final newline,
  trim trailing whitespace (sauf en Markdown où les espaces de fin
  sont des sauts de ligne). Match les conventions déjà en place
  dans le code, supprime les frictions entre éditeurs.
- [package.json] (new, repo root) Devient le package racine pour
  l'outillage partagé. Devdeps : husky, lint-staged, prettier.
  `lint-staged` configuré pour : Prettier sur \*.{json,md,yml,yaml},
  `npm run lint -- --fix` sur le sous-package backend ou frontend
  selon le fichier staged.
- [.prettierrc.json] (new) Config Prettier : single quotes, semis,
  2 espaces, trailing comma ES5, printWidth 100, arrow parens always,
  LF.
- [.prettierignore] (new) Tous les arbres `src/` de backend et
  frontend sont **ignorés par Prettier** — pas de reformatage de
  masse. Seuls les fichiers config/docs au niveau racine et dans
  doc/ sont formatés. Les fichiers source restent gérés par ESLint
  --fix.
- [.husky/pre-commit] (new) Lance `npx --no-install lint-staged`
  à chaque commit. Avant le hook, `git commit --no-verify` permettait
  juste de skipper les vérifs ; maintenant le hook les déclenche
  automatiquement et `--no-verify` reste l'échappatoire pour les
  urgences.
- [package-lock.json] (new, repo root) Lockfile pour les devdeps
  racine.
- [doc/STYLE_GUIDE.md] Mise à jour §11 Lint & format : Prettier
  et `.editorconfig` ne sont plus "absent / not wired up". Ajout
  d'une ligne Husky + lint-staged.
- [CONTRIBUTING.md] §5 mis à jour : précise que le pre-commit hook
  existe maintenant, comment l'activer (`npm install` à la racine),
  et comment le bypasser en urgence (`--no-verify`).

## Vague 1 — sécurité (lot A : env-time guards)

- [backend/.env.example] (new) Le fichier n'existait pas malgré les
  références dans CONTRIBUTING.md / backend/doc/. Créé avec des
  valeurs d'exemple anonymisées : `admin@example.com`,
  `change-me-in-production-please` pour le mot de passe admin, et
  `replace-with-a-long-random-string` pour SESSION*SECRET. Tous
  les env vars utilisés par le code y sont listés et commentés
  (NODE_ENV, DB*_, REDIS\__, SESSION*\*, ADMIN*_, FRONTEND*URL,
  RATE_LIMIT*_, LOG_LEVEL).
- [backend/src/utils/setup.js] AUTH-PROD-GUARD (chantier 1.2) :
  refuse explicitement de seeder un admin en production si
  ADMIN_PASSWORD est absent, vaut la valeur par défaut documentée
  (`change-me-in-production-please`), ou vaut l'ancien défaut
  legacy (`VortexFlow2024!`). En dev/test rien ne change — les
  tests d'auth peuvent continuer avec une valeur déterministe.
- [backend/tests/unit/utils/setup.test.js] (new) 6 tests pour
  AUTH-PROD-GUARD : refus en prod sans ADMIN_PASSWORD, refus en
  prod avec le default actuel, refus en prod avec le legacy
  default, acceptation en prod avec un mot de passe non-default,
  acceptation en dev avec le default, acceptation en dev sans
  ADMIN_PASSWORD (fallback). Mock du modèle User pour ne pas
  toucher la DB.
- [backend/src/config/env.js] (new, chantier 1.4) Validation joi
  de process.env au boot. Schéma : NODE*ENV (enum), PORT/HOST,
  DATABASE_URL ou DB** individuels, REDIS_URL/REDIS_PASSWORD,
  SESSION_SECRET (>=16 chars *requis* en prod, default dev sinon),
  ADMIN*EMAIL (email valide), FRONTEND_URL (URI), RATE_LIMIT**,
  LOG_LEVEL. Tolère les vars inconnues (CI / platforms).
  Mutate l'objet env pour appliquer les defaults, le code en aval
  continue de lire process.env.X.
- [backend/tests/unit/config/env.test.js] (new) 9 tests :
  acceptation minimal dev, refus prod sans SESSION_SECRET, refus
  prod si SESSION_SECRET trop court, acceptation prod si long,
  refus PORT non numérique, refus REDIS_URL malformé, refus
  ADMIN_EMAIL malformé, acceptation des vars inconnues,
  application des defaults au paramètre.
- [backend/server.js] Wire `require('./src/config/env').validate()`
  juste après `dotenv.config()`. Toute config invalide fait planter
  le boot avec un message lisible plutôt qu'une erreur runtime
  opaque plus tard.

Chantier 1.5 (rate-limit granulaire login/register) : déjà en
place. `middleware/auth.js::authRateLimit` impose 5 tentatives par
15 min en prod, 20 par 5 min en dev. Appliqué sur `/api/auth/login`
et `/api/auth/register` via le router. Aucune modification
nécessaire.
